IBM i Save Concepts — Four Areas You Must Cover

IBM i backup is different from file-system backup on Linux or Windows. You are not backing up a directory tree — you are saving a collection of typed objects that live in a hierarchical namespace: libraries containing objects (files, programs, data areas, job queues), the Integrated File System (IFS) directory tree, system configuration (network attributes, system values, device descriptions), and licensed internal code (LIC). Missing any of these four areas produces a backup that cannot fully restore a production system.

  • Libraries and objects — the primary IBM i namespace. Every DB2 table, RPG program, CL program, data area, output queue, and user profile lives in a library. Saved with SAVLIB and SAVOBJ.
  • IFS (Integrated File System) — the Unix-style directory tree. Holds Java applications, Python scripts, web content, PASE binaries, and XML/JSON data files. Saved with SAV.
  • System configuration — network attributes, line descriptions, controller descriptions, device descriptions, subsystem descriptions, user profiles, authorisation lists. Saved with SAVSYS or option 22 of GO SAVE.
  • Licensed Internal Code — the lowest layer of the OS. Only saved as part of a full system save (GO SAVE option 21). Cannot be saved independently in a useful way for restore purposes.

IBM i save commands are object-aware: they understand object types, lock the objects during save (with options to limit disruption), record the save date in the object header, and can save only changed objects since the last save. This is fundamentally more capable than block-level or file-system snapshots.

GO SAVE — Full, System Data, and User Data

The GO SAVE menu (type GO SAVE at a command line) is the traditional starting point for IBM i backup. It provides three primary options:

  • Option 21 — Save the entire system: Saves LIC, operating system, all system libraries (*NONSYS), all user libraries, and the IFS. Requires all users to be signed off and all subsystems except the controlling subsystem to be ended. The system must be in a restricted state. An option 21 save is the only save from which you can perform a bare-metal restore of a completely blank system.
  • Option 22 — Save system data only: Saves LIC, the operating system (QSYS and related system libraries), and system configuration. Does not save user libraries or IFS data. Used when you want to update OS PTFs and want a restore point for system data only.
  • Option 23 — Save all user data: Saves all user libraries (*ALLUSR), configuration objects, and user portions of the IFS (excludes /QSYS.LIB which is covered by SAVLIB). Does not save LIC or the operating system. This is the option used for routine scheduled backups on systems where the OS is stable.

Initiate an option 21 save to tape device TAP01:

GO SAVE
/* Select option 21, specify device TAP01, answer prompts */

Or automate it unattended with SAVSYS + SAVLIB(*NONSYS) + SAV sequence in a CL program. GO SAVE option 21 effectively runs this sequence for you with IPL-required locked state.

Check when the last full system save occurred:

SELECT
    OBJECT_LIBRARY,
    OBJECT_NAME,
    OBJECT_TYPE,
    LAST_SAVE_TIMESTAMP,
    LAST_SAVE_SIZE_KILOBYTES,
    LAST_SAVE_VOLUME_ID
FROM QSYS2.OBJECT_STATISTICS
WHERE OBJECT_LIBRARY = 'QSYS'
  AND OBJECT_NAME    = 'QSYS'
  AND OBJECT_TYPE    = '*LIB';

SAVLIB — Saving Libraries

SAVLIB is the workhorse command for saving individual libraries or groups of libraries. It is used for scheduled backups of user libraries without requiring a full system restricted state.

Save a single library to a save file (useful for automated offsite transfer):

SAVLIB LIB(ORDLIB)
       DEV(*SAVF)
       SAVF(QGPL/ORDLIB230601)
       TGTRLS(*CURRENT)
       UPDHST(*YES)
       OMITOBJ((*ALL *DTAQ) (*ALL *USRSPC))

UPDHST(*YES) records the save date in each saved object's header — essential for incremental saves later. OMITOBJ here excludes data queues and user spaces which are transient and do not need to be backed up.

Save all user libraries to a tape device:

SAVLIB LIB(*ALLUSR)
       DEV(TAP01)
       ENDOPT(*REWIND)
       UPDHST(*YES)
       PRECHK(*YES)

PRECHK(*YES) checks that all libraries can be saved before starting, preventing a partial save that fails midway through.

Omit specific libraries from *ALLUSR (e.g., libraries used only for work files):

SAVLIB LIB(*ALLUSR)
       DEV(TAP01)
       OMITLIB(WRKLIB TMPLIB STAGLIB)
       UPDHST(*YES)

Save only changed objects (incremental) using SAVCHGOBJ — objects modified since a reference date:

SAVCHGOBJ OBJ(*ALL)
           LIB(ORDLIB)
           DEV(*SAVF)
           SAVF(QGPL/ORDLIBCHG)
           REFDATE('2026-06-01')
           REFTIME('000000')
           UPDHST(*YES)

This saves only objects in ORDLIB whose last-change date is after midnight on 1 June 2026. Combined with a periodic full SAVLIB, this forms the basis of an incremental backup strategy.

Parallel save using multiple tape devices (dramatically faster for large libraries):

SAVLIB LIB(*ALLUSR)
       DEV(TAP01 TAP02 TAP03)
       UPDHST(*YES)
       PRECHK(*YES)

IBM i distributes the parallel tape streams automatically across the specified devices.

SAVOBJ and SAVSRVPGM — Object-Level Saves

SAVOBJ saves individual objects within a library. This is useful when you want to save a specific program before applying a change, or when a library is too large to save fully during a maintenance window.

Save a specific RPG service program before patching it:

SAVOBJ OBJ(ORDCALCR)
        LIB(ORDLIB)
        DEV(*SAVF)
        SAVF(QGPL/ORDCALCRBK)
        OBJTYPE(*SRVPGM)
        UPDHST(*YES)

Save all programs and service programs in a library:

SAVOBJ OBJ(*ALL)
        LIB(ORDLIB)
        DEV(TAP01)
        OBJTYPE(*PGM *SRVPGM)
        UPDHST(*YES)

Save all object types in a library (equivalent to SAVLIB for that library, but with more filtering options):

SAVOBJ OBJ(*ALL)
        LIB(ORDLIB)
        DEV(*SAVF)
        SAVF(QGPL/ORDLIBFULL)
        OBJTYPE(*ALL)
        UPDHST(*YES)
        TGTRLS(*CURRENT)

Save a user profile and its associated private authorities:

SAVSECDTA USRPRF(JSMITH)
           DEV(*SAVF)
           SAVF(QTEMP/JSMITHSEC)

SAV for IFS — Saving the Integrated File System

The IFS directory tree is saved with the SAV command, not SAVLIB. The IFS holds Python scripts, Java applications, PASE binaries, web server content, node.js applications, and any data files written by open-source tools running in PASE.

Save selected IFS paths to a save file:

SAV DEV('/QSYS.LIB/QGPL.LIB/IFSBK230601.FILE')
    OBJ(('/home')
        ('/QOpenSys/QIBM/UserData')
        ('/www')
        ('/opt'))
    SUBTREE(*ALL)
    UPDHST(*YES)

Note that /QSYS.LIB is the IFS projection of the IBM i library namespace. You do NOT save it with SAV — it is already covered by SAVLIB. Saving it twice wastes tape and time.

Save the entire IFS except QSYS.LIB and temporary paths:

SAV DEV('/QSYS.LIB/QGPL.LIB/IFSFULL.FILE')
    OBJ(('/*' *INCLUDE)
        ('/QSYS.LIB' *OMIT)
        ('/QDLS' *OMIT)
        ('/tmp' *OMIT)
        ('/QIBM/ProdData' *OMIT))
    SUBTREE(*ALL)
    UPDHST(*YES)

/QIBM/ProdData contains IBM product data that is restored by PTF install; you generally omit it from user data saves.

Save changed IFS objects only (incremental):

SAV DEV('/QSYS.LIB/QGPL.LIB/IFSCHG.FILE')
    OBJ(('/home' *INCLUDE)
        ('/www' *INCLUDE))
    SUBTREE(*ALL)
    CHGPERIOD('2026-06-01' '000000' *NOW *NOW)
    UPDHST(*YES)

RSTLIB and RSTOBJ — Restoring

Restore is the purpose of save. Understanding restore sequence and dependencies is as important as understanding save commands.

Restore a full library from a save file:

RSTLIB SAVLIB(ORDLIB)
        DEV(*SAVF)
        SAVF(QGPL/ORDLIB230601)
        RSTLIB(ORDLIB)
        MBROPT(*ALL)
        ALWOBJDIF(*ALL)

MBROPT(*ALL) restores all members of physical files, including members added after the last save. ALWOBJDIF(*ALL) allows object differences between the save media and the current system — necessary when restoring to a different system or after an OS upgrade.

Restore a specific object:

RSTOBJ OBJ(ORDCALCR)
        SAVLIB(ORDLIB)
        DEV(*SAVF)
        SAVF(QGPL/ORDCALCRBK)
        OBJTYPE(*SRVPGM)
        RSTLIB(ORDLIB)

Restore IFS objects:

RST DEV('/QSYS.LIB/QGPL.LIB/IFSBK230601.FILE')
    OBJ(('/home/*') ('/www/*'))
    SUBTREE(*ALL)

Restore sequence dependencies matter. When restoring a complete library that contains RPG programs bound to service programs:

  1. Restore the library structure first (RSTLIB with data-only objects).
  2. Restore service programs (*SRVPGM) before the programs (*PGM) that bind to them.
  3. Restore physical files (*FILE with FILE(*PHY)) before logical files (*FILE with FILE(*LGL)) that are based on them.
  4. Restore programs last.

IBM i handles most of this automatically when restoring a complete library save, but partial restores require attention to dependency order.

Virtual Optical Media with IMGCLG

Virtual optical media (image catalogues) allow you to save to and restore from virtual disc images on the IFS instead of physical tape. This is the modern approach for environments without tape infrastructure and for cloud-offload scenarios where save files are uploaded to object storage after creation.

Create an image catalogue:

CRTIMGCLG IMGCLG(ORDLIBCLG)
           DIR('/imgclg/ordlib')
           TYPE(*OPT)
           TEXT('ORDLIB monthly save image catalogue')

Add virtual media entries to the catalogue:

ADDIMGCLGE IMGCLG(ORDLIBCLG)
            FROMFILE(*NEW)
            TOFILE(ORDLIBVOL001)
            IMGSIZ(4700)

IMGSIZ(4700) creates a 4.7 GB virtual disc image — equivalent to a DVD. For large libraries, add multiple entries; IBM i spans them automatically.

Load (mount) the image catalogue so save commands can use it:

LODIMGCLG IMGCLG(ORDLIBCLG)
           DEV(IMGCLGDEV01)

Now use the virtual device IMGCLGDEV01 in a SAVLIB command exactly as you would use TAP01:

SAVLIB LIB(ORDLIB)
       DEV(IMGCLGDEV01)
       UPDHST(*YES)
       ENDOPT(*UNLOAD)

After saving, the image files reside in /imgclg/ordlib/ on the IFS. You can then upload them to IBM Cloud Object Storage, AWS S3, or Azure Blob Storage using any PASE tool:

/* From PASE — upload completed image to S3 */
QSH CMD('aws s3 cp /imgclg/ordlib/ORDLIBVOL001 \
    s3://mybucket/ibmi-backups/2026-06-01/ \
    --region eu-west-2')

To restore from a cloud-stored image, download it back to the IFS, re-add it to an image catalogue with ADDIMGCLGE, load it with LODIMGCLG, and restore with RSTLIB pointing to the virtual device.

BRMS — Backup Recovery and Media Services

BRMS (5770-BR1) is IBM's integrated backup automation and media management product for IBM i. It transforms the manual CL commands described above into a policy-driven, scheduled, and tracked system. BRMS is the right choice for any production IBM i where backup must be reliable, auditable, and recoverable without manual intervention.

What BRMS adds over manual CL commands:

  • Media management — tracks every tape or virtual media volume: what is on it, when it expires, where it is (on-site, off-site, in-transit). WRKMEDIBRM displays the media inventory.
  • Backup policies (control groups) — define what to save, in what order, to what device, on what schedule. A control group replaces a hand-crafted CL backup program.
  • Incremental and cumulative tracking — BRMS tracks save history per object and per library. It knows which objects have changed since the last full or incremental save without you having to manage REFDATE parameters manually.
  • Recovery reports — BRMS generates step-by-step recovery instructions for any restore scenario based on current media inventory.
  • Integration with system journal — BRMS logs all backup activity to the system journal for audit purposes.

Start a BRMS backup using a defined backup control group:

STRBKUBRM CTLGRP(NIGHTLY_USR)
           STRSEQ(*STRSEQ)

Start a BRMS recovery (restores from BRMS-managed media):

STRRCYBRM CTLGRP(*SYSTEM)
           OPTION(*RESTORE)

Display media inventory:

WRKMEDIBRM VOL(*ALL)
            MEDCLS(*ALL)
            EXPIRED(*NO)

Move media to off-site (update location tracking):

MOVMEDBRM FROMLOC(*HOME)
           TOLOC(OFFSITE1)
           MEDCLS(*ALL)

Display what is saved on a specific volume:

DSPSAVBRM VOL(TAPVOL001)
           OUTPUT(*PRINT)

Create a new backup control group for nightly user library saves:

CRTCTLGBRM CTLGRP(NIGHTLY_USR)
            TYPE(*BKU)
            TEXT('Nightly user library incremental')

Then add backup items to the control group:

ADDLBRBRM CTLGRP(NIGHTLY_USR)
           SEQ(10)
           LIB(*ALLUSR)
           OMITLIB(WRKLIB TMPLIB)
           SAVACT(*SYNCLIB)
           UPDHST(*YES)
           INCLIB(*YES)

INCLIB(*YES) tells BRMS to save only objects changed since the last save recorded in BRMS history — making this an incremental backup automatically.

Backup Schedule Design

A practical IBM i backup schedule uses three tiers that match recovery point objectives with media costs and backup window constraints.

Daily incremental — SAVCHGOBJ

Run nightly, typically in a maintenance window between 22:00 and 02:00. Save only changed objects using SAVCHGOBJ or a BRMS incremental control group. This produces small, fast saves that can complete within an hour even for large systems.

/* CL for nightly incremental — run from a scheduled job */
SAVCHGOBJ OBJ(*ALL)
           LIB(*ALLUSR)
           OMITLIB(WRKLIB TMPLIB)
           DEV(*SAVF)
           SAVF(QGPL/INCR0620)
           REFDATE(*SAVLIB)
           UPDHST(*YES)

REFDATE(*SAVLIB) uses the recorded last-save date from the object's save history as the reference point.

Weekly differential — SAVLIB

Run once a week (typically Sunday night) saving all user libraries in full. This is the baseline from which daily incrementals apply forward. Rotate media so that two weekly saves are available before the oldest is reused.

SAVLIB LIB(*ALLUSR)
       DEV(TAP01)
       OMITLIB(WRKLIB TMPLIB)
       ENDOPT(*REWIND)
       UPDHST(*YES)
       PRECHK(*YES)

Monthly full — GO SAVE option 21

Run once a month in a planned maintenance window (requiring an IPL to restricted state). This is the only save from which a bare-metal restore is possible. Store monthly full saves off-site. Retain at least three monthly saves before recycling media.

The recovery sequence from a complete system loss:

  1. Install LIC from D-mode IPL (distribution tape or virtual media).
  2. Restore the operating system from the most recent option 21 full save.
  3. Restore system configuration (SAVSYS data).
  4. Restore user libraries from the most recent weekly full save (RSTLIB *ALLUSR).
  5. Apply daily incrementals in sequence from the weekly save date forward (RSTOBJ or BRMS STRRCYBRM).
  6. Restore IFS from the most recent SAV save.
  7. Verify by testing key application functions before handing back to users.

Test your restore procedure at least quarterly. An untested backup is not a backup.

Summary

IBM i save and restore is a layered discipline: GO SAVE option 21 for full system bare-metal recovery, SAVLIB for library-level saves, SAVOBJ for object-level precision, SAV for IFS, and IMGCLG virtual optical media for tapeless cloud-offload. BRMS transforms these manual commands into a managed policy system with media tracking, incremental history, and recovery reports. A well-designed schedule — daily SAVCHGOBJ, weekly SAVLIB, monthly GO SAVE 21 — combined with off-site storage and quarterly restore tests gives IBM i the reliable backup posture the platform's business-critical workloads demand.
